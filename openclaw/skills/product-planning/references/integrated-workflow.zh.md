# Product-Planning V1 整合流程

本流程保留 V1 的 7 个 Sheet、版式和会议表达方式，同时加入统一证据对象、查询口径锁定、数据一致性校验和输出质量门禁。所有 Markdown 与 Excel 内容必须从同一份 `evidence.json` 生成。

## 一、总体架构

```text
三款种子 ASIN
  -> 输入与凭证预检
  -> ASIN/站点/相关性校验
  -> 三款种子 + 一款系统补充竞品
  -> 类目候选评分与目标节点锁定
  -> 固定口径调用 SellerSprite MCP
  -> evidence.json 统一证据对象
  -> Python 规则校验
  -> Markdown + V1 七 Sheet Excel
  -> 文件、图表、占位符和 XML 校验
  -> 脱敏摘要 Gemini + GLM 交叉验证
  -> 进入 / 观察 / 放弃
```

## 二、不可变规则

1. 命令固定为 `/product-planning "{ASIN1;ASIN2;ASIN3}" {SITE}`，三个 ASIN 必须唯一、有效且属于指定站点。
2. 数据优先级为 SellerSprite MCP、Amazon 前端补充证据、用户提供的数据或假设。MCP 空字段写 `N/A`，不得推断补值。
3. 三款用户竞品必须保留；系统只补充第四款。没有合格候选时第四款写 `N/A`。
4. 市场分析必须使用已锁定类目节点的数据，禁止用 ASIN 汇总冒充类目数据。
5. 当前未完整月份只用于快照；12 个月趋势应优先采用完整月份，并在 `queryManifest` 中写明口径。
6. Markdown 是主报告；Excel 是额外会议版输出。二者必须引用同一证据对象。
7. 默认只生成 7 个 V1 Sheet，不生成成本试算和销售计划。

## 三、阶段与门禁

### 阶段 0：输入与凭证预检

- 解析三个以分号分隔的 ASIN 和站点。
- `GB` 在 MCP 调用前归一为 `UK`。
- 缺少 SellerSprite 凭证时停止真实数据采集并向用户索要。
- 仅在用户要求真实双模型验证且缺少模型凭证时索要 Gemini/GLM key。

**门禁：** ASIN 数量不是 3、重复、格式错误或站点不支持时不得进入下一阶段。

### 阶段 1：种子竞品与产品方向

对每个种子调用 `asin_detail`，核验：

- ASIN 存在性与站点；
- 标题、功能、场景、目标用户和兼容对象；
- `nodeIdPath` 与类目路径；
- 价格、评分、评分数、品牌、履约、上架时间和 Listing 特征。

从三款种子的共同证据推导产品方向，不以用户未提供的工程或供应链信息补全产品定义。

### 阶段 2：第四竞品补充

用 `competitor_lookup` 和 `product_research` 建立候选池，再逐个用 `asin_detail` 校验。优先补齐三款种子未覆盖的一个角色：

- 头部评价壁垒；
- 低价冲击；
- 高配功能差异化。

候选必须记录纳入或排除原因。第四款无法通过校验时保持 `N/A`，不得用搜索噪声替代。

### 阶段 3：类目节点评分与锁定

从四款已验证竞品的 `nodeIdPath` 生成候选节点，并按以下顺序决策：

1. 优先选择重复出现且与产品形态直接一致的最深叶子节点；
2. 若无重复叶子节点，评估覆盖多数种子的最小父节点；
3. 父节点包含明显不同产品形态、价格结构过宽或购买意图混杂时判为过宽；
4. 可保留一个直接相关的相邻节点作交叉验证，但不能混合成主市场样本。

`categorySelection.selected` 必须保存节点路径、名称路径、覆盖率、相关性、选择理由和被拒绝候选。

### 阶段 4：固定口径采集

在调用市场接口前先写入 `queryManifest`：

| 参数 | 规则 |
|---|---|
| `analysisMonth` | 当前数据快照月份，格式 `YYYYMM` |
| `lastCompleteMonth` | 最近完整月份 |
| `trendMonths` | 连续 12 个月，按升序保存 |
| `topN` | 市场样本口径，默认 100 或接口实际值 |
| `newProductMonths` | 新品定义，默认 6 个月或接口实际值 |
| `marketplace` | MCP 实际调用站点 |
| `selectedNodeIdPath` | 阶段 3 锁定的主节点 |

固定调用顺序：

1. `market_research_statistics`：连续 12 个月的类目均销、均销售额、平均 BSR 等；
2. `market_product_demand_trend`：需求快照、退货或搜索购买指标；空返回时只按文档允许的月份参数重试，并记录重试；
3. 价格、评分值、评分数、上架时间和上架年份分布；
4. `keyword_miner` 与关键词趋势工具；
5. `review`：4-5 星卖点和 1-3 星痛点；
6. ABA/流量工具、品牌/卖家集中度、Google Trend 或商标工具作为可选补充。

每次调用要保存工具名、脱敏入参摘要、查询时间、返回行数、采用状态、缺失字段和 `evidenceId`。

### 阶段 5：统一证据对象

`evidence.json` 至少包含：

- `schemaVersion`、`command`、`queryManifest`；
- `productDirection`、`competitors`；
- `categorySelection`；
- `market.monthlyTrend` 与各类分布；
- `keywords`、`voc`；
- `dataSources`、`dataGaps`；
- `decision.gates`、`decision.actions`；
- `render.sheetOrder` 和输出标题。

数值保留原始数值类型；展示格式、百分号和货币符号在渲染层处理。每个关键结论与行动必须引用一个或多个 `evidenceId`。

### 阶段 6：Python 规则校验

在渲染报告前运行：

```text
python scripts/python/validate_product_planning_evidence.py evidence.json --output evidence_qa.json
```

`FAIL` 阻止生成正式报告；`WARN` 可以生成，但必须进入数据缺口和验证摘要；`PASS` 才表示结构与核心口径完整。

校验重点：

- 三款种子、四款最终竞品及校验状态；
- 类目节点覆盖率和选择理由；
- 12 个月连续趋势和完整月份口径；
- 关键词请求数、返回数和缺失项一致；
- 分布占比合计、上架时间与上架年份异常；
- VOC 样本量及“样本不足”标记；
- 决策门禁和行动的证据引用；
- 模板占位符、密钥、绝对路径和敏感 URL。

### 阶段 7：双格式渲染

V1 Sheet 顺序固定：

1. `意向产品`
2. `市场分析`
3. `竞品分析与优化策略`
4. `SWOT分析`
5. `ABA排名【季度】`
6. `MCP原始数据`
7. `数据来源`

各 Sheet 的数据职责：

| Sheet | 主要证据 |
|---|---|
| 意向产品 | 已验证竞品共同功能、场景、用户、前端主图 |
| 市场分析 | 目标节点 12 月趋势、关键词和各类分布 |
| 竞品分析与优化策略 | 三款种子、第四竞品、Listing、正负评论 |
| SWOT分析 | 前三 Sheet 的证据重组，不新增事实 |
| ABA排名【季度】 | 空白人工维护页；自动流程清除内容与图表，不写入接口数据 |
| MCP原始数据 | 脱敏后的紧凑原始或聚合数据 |
| 数据来源 | 工具、入参口径、时间、采用状态、缺口和证据 ID |

### 阶段 8：输出质量与模型验证

生成后必须执行：

1. Markdown 与 Excel 标题、文件名一致；
2. 全局无 `{{...}}`、`TODO` 或零值假占位；
3. 图表引用到实际数据区域，空数据不生成误导性图表；
4. `.xlsx` Office XML 完整性通过；可用桌面 Excel 时确认无修复提示；
5. 对报告生成脱敏摘要，仅发送聚合数据、口径、结论和证据引用给 Gemini/GLM；
6. 将规则校验和双模型验证结果写入验证摘要，但不覆盖原始 MCP 事实。

## 四、决策输出

最终结论只能是 `进入`、`观察` 或 `放弃`。每个门禁包含状态、阈值、实际证据和 `evidenceId`。产品经理行动最多保留 P0/P1/P2 三层，并优先覆盖：

- 样品与兼容性验证；
- Listing 卖点表达；
- 定价或广告投放验证。

缺少供应链、成本、专利或合规证据时，只能登记为待验证项，不能据此直接给出确定性结论。
